🎯 这个 bench 在测什么

这是一个时序 caption 的「可区分性 (distinctiveness)」评测集。一句好的时序 caption,必须说出这个片段独有的内容,而不是泛泛地把 query 复述一遍。

  • 每个组 (group) = 一个视频 + 一个共享 query,这个 query 在该视频里出现在 K 个不同片段。
  • 这 K 个同 query 片段是天然的强干扰项:它们都满足 query,只在细节上不同 → 逼模型写出区分性。

🤖 Gemini 预标了什么(你来核验)

  • ★ specific = 该段独有的事实(区分性的关键)。
  • · shared = 全组共有的事实(≈ query 本身)。
  • negatives = 看似合理、但画面里其实不存在的描述。
这些只是机器预标,正式评分不依赖任何 LLM 裁判 —— 全靠你这一步人工核验把关。

①结构正确性(决定整组去留)

  • 每段 · query 是否真的在此段出现? Gemini 偶尔会选错段 → 选「否」剔除。
  • 整组 · 各段是否是不同的独立时刻? 若其实是同一次发生被切成几段,选「否」。
  • 据此给整组结论:✓ 保留 / ✗ 丢弃。

②探针质量(逐段三项)

  • facts 是否与画面相符(对 / 有问题)。
  • specific 标签是否合理:标 specific 的确实是本段独有,而不是全组都有。
  • negatives 是否确实不存在于画面。
  • 发现个别错的 → 直接点那条 fact / negative 文本把它划掉标记。

🖱 怎么操作

  • 左侧选组;视频可播放 / 循环反复看。
  • 点 chip 选「是/否」「对/有问题」;点 fact/negative 文本标错。
  • 填完每段①+②三项、组①(是否不同+保留/丢弃)后,点绿色「标记本组完成」自动跳下一组。
  • 自动保存(右上圆点:绿=已存)。快捷键: Ctrl+Z 撤销 · J/K 上/下一组

💡 判断小贴士

  • 该 reject 的:query 根本没出现 / K 段其实是同一次 occurrence / 片段损坏看不清。
  • specific 标错:把全组都有的东西标成了 specific。
  • negative 标错:写的「不存在」其实画面里有。
  • 左侧带 ⚠ 的组是 Gemini 自动标记的可疑组,优先重点看。
